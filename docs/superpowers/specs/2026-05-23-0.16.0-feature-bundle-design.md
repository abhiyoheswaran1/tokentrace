# TokenTrace 0.16.0 — Feature Bundle Design

Date: 2026-05-23
Status: Approved scope, building.

Ships three independent feature areas as one release. Each feature lands as a
focused commit (or small commit train) on `main`. Each commit verifies
independently before the next begins. The release commit only bumps the
version, updates the changelog, and pins the metadata files.

## Goals

- Give users direct, local action on parser misclassifications.
- Give users (and agents) saved, replayable reports without re-filtering by hand.
- Give multi-agent workflows a structured way to pass state between agents.

## Non-goals

- No cloud features, no sharing service, no telemetry.
- No PDF export in 0.16.0 (HTML and Markdown only).
- No remote agent invocation; handoff is read-only data the next agent fetches.

## Feature 1: Parser Studio

### Problem

The `/parser-debug` page shows what the parser decided per file but the user
cannot act on a misclassification. Import Profiles in Settings cover
folder-level rules, not file-level overrides.

### Scope

1. **Per-file parser override** — mark a file as `use parser X` or `exclude`.
   Persisted locally, applied on the next scan.
2. **Why-this-parser view** — surfaces matcher hit, confidence, ranked
   candidates, and override status inline in `/parser-debug`.
3. **Reparse-and-diff preview** — predicts what an alternate parser would
   produce for one file. Read-only; no DB writes.

### Data model

New table `file_parser_overrides`:

| column      | type    | notes                                  |
|-------------|---------|----------------------------------------|
| path        | TEXT PK | absolute file path                     |
| parser_id   | TEXT    | NULL if the override is `excluded`     |
| excluded    | INTEGER | 0 or 1                                 |
| note        | TEXT    | optional user note                     |
| created_at  | TEXT    | ISO-8601                               |
| updated_at  | TEXT    | ISO-8601                               |

Migration added to `src/db/migrate-core.ts`. Additive only.

### Modules

- `src/lib/parser-overrides.ts` — pure CRUD over the new table.
- `src/ingestion/parser-resolver.ts` — wraps existing parser detection. If an
  override exists for the path, it short-circuits.
- `app/api/parser-overrides/route.ts` — `POST` to upsert, `DELETE` to clear.
- `app/api/parser-debug/preview/route.ts` — `POST` `{ path, parserId }` returns
  predicted sessions, interactions, tokens, cost. Never writes.
- UI: kebab menu in `/parser-debug` row → "Override parser…" modal; "Why?"
  expander row beneath each file; "Try parser X" button inside the Why panel.

### CLI

`tokentrace repair set-parser <path> --parser <id>` and
`tokentrace repair set-parser <path> --exclude [--note "…"]`.
`tokentrace repair clear-parser <path>`.

### Tests

- `tests/parser-overrides-crud.test.ts` — table operations.
- `tests/parser-overrides-ingestion.test.ts` — override short-circuits parser
  detection during scan.
- `tests/parser-debug-preview-api.test.ts` — preview endpoint returns predicted
  shapes; never persists.
- `tests/parser-overrides-cli.test.ts` — `tokentrace repair set-parser` happy
  path + error paths.
- `tests/parser-debug-why.test.tsx` — expander row renders matcher and
  candidates.

### Risk and rollback

- Override table is additive; absence means default behavior. Rollback is
  truncating the table.
- Preview endpoint is read-only; cannot corrupt local data.

## Feature 2: Reports

### Problem

Evidence Packs cover a single view. Recurring questions (weekly cost, monthly
top models, per-project drift) require manual filtering each time.

### Scope

1. **Saved reports** — turn the current dashboard view (period + filters) into a
   named, persisted report.
2. **`tokentrace report --saved <name>`** — replay a saved report from the CLI
   in `json`, `markdown`, or `html`.
3. **Comparison reports** — built-in period-vs-prior-period and model-vs-model
   views.

### Data model

New table `saved_reports`:

| column        | type    | notes                                            |
|---------------|---------|--------------------------------------------------|
| id            | TEXT PK | nanoid                                           |
| name          | TEXT    | unique per local install                         |
| view_type     | TEXT    | `overview` / `sessions` / `models` / `comparison`|
| params        | TEXT    | JSON: period, filters, columns                   |
| format        | TEXT    | default `markdown`; one of json/markdown/html    |
| created_at    | TEXT    | ISO-8601                                         |
| last_run_at   | TEXT    | ISO-8601, nullable                               |

Migration added to `src/db/migrate-core.ts`. Additive only.

Unique constraint on `name` enforced at the application layer (case-insensitive)
so the CLI can refer to reports by name.

### Modules

- `src/lib/saved-reports.ts` — CRUD + name-uniqueness + param validation
  (whitelist of allowed filter keys).
- `app/api/saved-reports/route.ts` — list / create.
- `app/api/saved-reports/[id]/route.ts` — get / update / delete.
- `app/reports/page.tsx` — list view with run buttons.
- `app/reports/compare/page.tsx` — comparison report viewer.
- `scripts/report.ts` — accept `--saved <name>` and `--format html` flags.

HTML export is a stylesheet-less standalone page (inline CSS, no scripts) so it
opens correctly in any browser, archives cleanly, and has no XSS surface from
user-supplied filter values (all interpolation is escape-encoded).

### CLI

- `tokentrace report --saved <name> --format json|markdown|html [--out <path>]`
- `tokentrace report --list-saved [--json]`
- `tokentrace report compare --period-1 ISO..ISO --period-2 ISO..ISO [--model <id>] --format ...`

### Tests

- `tests/saved-reports-crud.test.ts` — CRUD + name uniqueness + invalid param
  rejection.
- `tests/report-saved-cli.test.ts` — `--saved <name>` replays correctly.
- `tests/report-comparison.test.ts` — period-vs-prior and model-vs-model math.
- `tests/report-html-escaping.test.ts` — HTML export escapes user-supplied
  filter values; no executable script tags.

### Risk and rollback

- Reports are stored locally; deleting the table removes them. No external
  state.
- HTML format is the only new surface for XSS-style bugs; covered by an
  explicit escaping test.

## Feature 3: Agent Handoff

### Problem

The MCP server lets one agent ask read-only questions but multi-agent
workflows (Claude does a scan, Codex picks it up later) have no structured way
to pass state between agents. There is also no observable record of what an
agent did locally.

### Scope

1. **Handoff envelope** — single structured JSON document an agent can request
   that summarizes current local state plus suggested next actions.
2. **MCP tool `get_handoff`** — same envelope served via MCP.
3. **Agent action log** — local rolling log of agent-initiated actions so the
   next agent has continuity.

### Data model

New table `agent_actions`:

| column           | type    | notes                                          |
|------------------|---------|------------------------------------------------|
| id               | INTEGER PK | autoincrement                               |
| ts               | TEXT    | ISO-8601                                       |
| surface          | TEXT    | `cli` or `mcp`                                 |
| command          | TEXT    | e.g. `scan`, `mcp:get_status`                  |
| outcome          | TEXT    | `ok` / `error`                                 |
| summary          | TEXT    | short human-readable line                      |
| payload          | TEXT    | JSON; bounded to 4KB                           |

Migration added to `src/db/migrate-core.ts`. Additive only.

Bounded retention: keep the last 500 entries. Trim happens on every write.
Configurable via `TOKENTRACE_AGENT_ACTION_LOG_MAX`.

### Modules

- `src/lib/agent-actions.ts` — append + trim + read (bounded by count or by
  time window) + redaction of obviously sensitive fields if any appear in the
  payload (defensive only; the call sites pass curated summaries).
- `src/lib/handoff.ts` — assembles the envelope. Pure read from local DB.
- CLI write hooks in `src/cli/commands.js` — record CLI command entry/exit.
- MCP write hooks in `src/lib/mcp-server.ts` — record tool calls.
- New MCP tool `get_handoff` in `src/lib/mcp/tools.ts` returning the envelope.
- `scripts/agent.ts` accepts `--handoff` and `--actions` flags.

### Handoff envelope shape (schema `tokentrace.handoff.v1`)

```json
{
  "$schema": "tokentrace.handoff.v1",
  "generatedAt": "2026-05-23T12:30:00.000Z",
  "scan": {
    "lastStartedAt": "...",
    "lastCompletedAt": "...",
    "filesScanned": 12,
    "recordsImported": 480
  },
  "repairQueue": {
    "unresolvedCount": 3,
    "topCauses": [{ "cause": "missing pricing", "count": 2 }],
    "focusHref": "/repair?key=…"
  },
  "confidence": {
    "overall": "medium",
    "drivers": ["estimated tokens for 12% of interactions"]
  },
  "recentActions": [
    { "ts": "...", "surface": "cli", "command": "scan", "outcome": "ok",
      "summary": "imported 480 records from 12 files" }
  ],
  "suggestedNextActions": [
    { "label": "Address top repair cause", "command": "tokentrace repair --focus 'missing pricing'" },
    { "label": "Refresh model rates", "command": "tokentrace doctor refresh-prices" }
  ]
}
```

### CLI

- `tokentrace agent --handoff [--json]` — envelope output.
- `tokentrace agent --actions [--json] [--limit N]` — last N action log rows.

### Tests

- `tests/agent-actions-bounded.test.ts` — write past the cap, oldest entries
  drop; configurable cap respected.
- `tests/agent-actions-redaction.test.ts` — payload that happens to contain a
  bearer-token-shaped string is redacted before write.
- `tests/agent-handoff-envelope.test.ts` — envelope shape matches the schema,
  no scan side effect, sensible suggested actions per current state.
- `tests/mcp-get-handoff.test.ts` — MCP tool returns the same envelope.
- `tests/agent-handoff-cli.test.ts` — `tokentrace agent --handoff --json`
  output validates.

### Risk and rollback

- Action log writes are best-effort; CLI exit hook swallows write errors so a
  log failure never breaks a CLI command.
- The log table is local and capped; cannot grow unboundedly.
- Handoff envelope is pure-read; no side effects.

## Cross-cutting decisions

### Commit shape

Three feature commits, then one release commit:

1. `feat(parser-studio): override + why + preview`
2. `feat(reports): saved reports + CLI + comparison view`
3. `feat(agent): handoff envelope + action log + get_handoff MCP tool`
4. `Release 0.16.0`

Each feature commit must pass: `npm test`, `tsc --noEmit`, `npm run lint`,
`npm run build:app` before the next begins.

### Migrations

Three new tables. All additive. Each gets a migration block in
`src/db/migrate-core.ts` guarded by the standard `duplicate column / table`
handling. Existing data is never touched.

### Test discipline

TDD per the superpowers methodology: every new module starts with a failing
test, then implementation. The existing test count (336 after 0.15.2) climbs;
the suite must stay green at every commit.

### Bug-hunt passes

Four explicit bug-hunt windows:

1. After Parser Studio — overrides race, SQL escaping in dynamic name fields,
   UI hydration in the modal, CLI argument edge cases.
2. After Reports — JSON params validation, HTML XSS, name-collision, CLI
   filename safety, comparison math edge cases.
3. After Agent Handoff — log unbounded growth, schema drift between CLI/MCP
   envelopes, sensitive data in payload, error swallowing.
4. Final cross-cutting — TODO/FIXME triage, dead code, `process.cwd()`
   fragility in `src/lib/mcp-server.ts:51`, `security:ioc` scan.

### Release ritual

- Bump `package.json` and `server.json` to `0.16.0`.
- Move `Unreleased` block of `CHANGELOG.md` into `## [0.16.0] - 2026-05-23`.
- Update `tests/changelog-next-version.test.ts` to assert the new version.
- `npm run release:check` must be green.
- Commit "Release 0.16.0", tag `v0.16.0`, push main + tag.
- GitHub workflow publishes npm + MCP registry via OIDC (no manual login).
- Update `docs/WEBSITE-UPDATE-PROMPT.md` with the 0.16.0 release message and
  hand the user a copy-paste prompt for the marketing site.

### Internal-roadmap-themes tests

`tests/roadmap-status.test.ts` and `tests/changelog-next-version.test.ts`
forbid the strings `"Parser Studio"`, `"Reports"`, and `"Agent Handoff"` from
the public changelog and roadmap-status. The 0.16.0 changelog entry uses
neutral phrasing ("parser overrides", "saved reports", "agent handoff
envelope") to stay compliant.
