# TokenTrace 0.17.0 Performance Bundle — Design

**Date:** 2026-05-23
**Status:** Approved (user authorized "GO AHEAD" on the three-slice proposal)
**Target release:** 0.17.0 (single bundled bump)

## Goal

Make the local dashboard feel measurably snappier. No behavior change, no UI change, no new features — pure performance work. User instruction: "improve performance and make the app work faster, feel snappier."

## Findings from exploration

- **SQLite pragmas are under-tuned on the runtime connection.** `src/db/client.ts` only sets `busy_timeout` and `foreign_keys`. `journal_mode=WAL` is set inside `migrate-core.ts` but never re-applied on subsequent connections; the cheap analytics pragmas (`cache_size`, `mmap_size`, `synchronous=NORMAL`, `temp_store=MEMORY`) are unset.
- **Indexes are already good.** Covering indexes exist on `interactions(analytics, session, cost_repair)` and `sessions(tool, project, started_at)`. Not the bottleneck.
- **Overview page is the heaviest path.** `app/page.tsx` is `force-dynamic` and on every request calls `getOverviewPageData`, which sequentially invokes: `getAnalyticsData`, `buildAccountingInvariants`, `buildPostSessionReview` (with `buildScanDiff`), `getDefaultSearchRoots`, `buildDoctorReport`, `buildUnknownCostRepairWorkbench`, `buildFirstRunStatus`. Each is independent SQL.
- **Statements aren't cached.** `analytics-query-helpers.ts:78` re-prepares SQL every call: `return sqlite.prepare(sql).all(...params) as T[]`. Hot endpoints pay the prepare+plan cost on every request.
- **Recharts is loaded eagerly.** `TrendSection` and `RankBarChart` are client components but not `next/dynamic`. Recharts adds ~80–100KB gzipped to the initial JS bundle for `/`.
- **Timing infrastructure exists.** `src/lib/analytics-timing.ts` already wraps queries with a configurable threshold (default 500ms) and stores slow-query samples. Dev/test only by default; can be flipped on with `TOKENTRACE_ANALYTICS_TIMING=1`.

## Non-Goals

- React Server Components streaming with `<Suspense>` (deferred to 0.18.0).
- Materialized views or cached aggregates (premature without baseline numbers).
- Bundle splitting beyond Recharts (audit deferred until 0.17.0 numbers land).
- Behavior changes, copy changes, schema changes.

## Architecture

Three independent, low-risk slices, all delivered together as 0.17.0.

### Slice 1 — DB connection tuning + statement cache

**What:** Adjust `src/db/client.ts` to set analytics-tuned pragmas on connection and introduce a tiny prepared-statement cache keyed by SQL string.

**Pragmas added (in this order):**

```
PRAGMA journal_mode = WAL;        -- safe re-apply; matches migrate-core
PRAGMA synchronous = NORMAL;      -- safe with WAL, ~2-3x faster commits
PRAGMA temp_store = MEMORY;       -- temp tables in RAM
PRAGMA cache_size = -65536;       -- 64MB page cache (negative = KB)
PRAGMA mmap_size = 268435456;     -- 256MB memory-mapped I/O
```

**Statement cache API:**

```ts
// src/db/prepared.ts (new)
export function prepareCached<T extends unknown[] = unknown[]>(sql: string): Database.Statement<T>;
export function clearPreparedCache(): void;  // for tests that reset the DB
```

Internally a `Map<string, Database.Statement>` on module scope. Cleared by `clearPreparedCache()` only — not auto-invalidated, because SQL strings are static.

**Wired into (concrete list, no rewrites of code that already prepares once and reuses):**
- `src/lib/analytics-query-helpers.ts` — `runAnalyticsQuery` (the hottest call site, called on every dashboard request)
- `src/lib/unknown-cost-repair/workbench.ts` — `runQuery`
- `src/lib/unknown-cost-repair/suggestions.ts` — `runQuery`
- `src/lib/scheduled-scan.ts` — last-scan lookup (called once per overview load)

Other `.prepare()` callers (`evidence/query.ts`, `pricing.ts`, `parser-trust.ts`, `session-timeline.ts`, `live-status.ts`, `accounting-invariants.ts`, `usage-guardrails.ts`, `saved-reports-store.ts`) are either already prepared-once-reused (kept as-is) or write paths called once per scan (not perf-critical).

**Expected impact:** 20–50% reduction in p50 query time on warmed cache; statement reuse eliminates parse+plan overhead.

### Slice 2 — Overview parallelization + per-request memo

**What:** Run the independent sub-queries in `getOverviewData` concurrently via `Promise.all`, and memoize within a single React server render using `React.cache`.

**Current shape (`src/lib/overview-data.ts`):**

```ts
const data = getAnalyticsData(...);
const accountingReport = buildAccountingInvariants(...);
const postSessionReview = buildPostSessionReview({ scanDiff: buildScanDiff(), ... });
const roots = await getDefaultSearchRoots();
const doctorReport = buildDoctorReport({ ...trust, roots });
const repairWorkbench = buildUnknownCostRepairWorkbench(...);
const firstRunStatus = buildFirstRunStatus({ ... });
```

**New shape:**

```ts
const [data, accountingReport, scanDiff, roots, repairWorkbench] = await Promise.all([
  Promise.resolve().then(() => getAnalyticsData(...)),
  Promise.resolve().then(() => buildAccountingInvariants(...)),
  Promise.resolve().then(() => buildScanDiff()),
  getDefaultSearchRoots(),
  Promise.resolve().then(() => buildUnknownCostRepairWorkbench(...)),
]);
// dependents stay sequential
const doctorReport = buildDoctorReport({ ...data.scanTrust, roots });
const postSessionReview = buildPostSessionReview({ scanDiff, ... });
const firstRunStatus = buildFirstRunStatus({ ..., latestScan: doctorReport.latestScan });
```

Wrap `getOverviewPageData` in `React.cache(...)` so duplicate calls within one render tree are deduplicated.

`better-sqlite3` is sync so true parallelism is bounded, but `Promise.all` lets the Node event loop interleave with the async work (`getDefaultSearchRoots` does filesystem I/O), which is the realized win.

**Expected impact:** overview TTFB drops by the cost of filesystem I/O overlap (~50–150ms on warm cache).

### Slice 3 — Lazy-load Recharts

**What:** Wrap chart components in `next/dynamic({ ssr: false })` with a skeleton fallback.

**Affected files:**
- `components/charts/trend-section.tsx`
- `components/charts/rank-bar-chart.tsx` (called from where it's used, not the leaf)

**Pattern:** Create a thin wrapper at the call site:

```tsx
// new wrapper, e.g., components/charts/trend-section-lazy.tsx
import dynamic from "next/dynamic";
import { ChartSkeleton } from "@/components/charts/skeleton";

export const TrendSection = dynamic(
  () => import("./trend-section").then((m) => m.TrendSection),
  { ssr: false, loading: () => <ChartSkeleton /> }
);
```

Add `components/charts/skeleton.tsx` if it does not exist already (matches the visual heft of the chart to avoid CLS).

**Expected impact:** first paint ~80KB lighter on `/`; charts appear after card content.

## Testing strategy (TDD per slice)

**Slice 1:**
- `tests/db-pragmas.test.ts` (new) — fresh DB, assert each pragma comes back with the expected value via `sqlite.pragma("name")`.
- `tests/prepared-statement-cache.test.ts` (new) — call `prepareCached(sql)` twice with same SQL, assert identical `Database.Statement` instance; assert distinct SQL → distinct statements; assert `clearPreparedCache()` works.
- Update `tests/analytics-query-helpers.test.ts` if present, or add it, to confirm the helper still returns correct rows after switching to cached prepare.

**Slice 2:**
- `tests/overview-data-parallel.test.ts` (new) — mock each sub-builder with a 50ms sleep. Assert total wall-clock for `getOverviewData` is well under the sum (i.e., parallelism realized).
- Existing overview tests must continue to pass unchanged (no behavior change).

**Slice 3:**
- `tests/lazy-charts.test.tsx` (new) — assert wrapper modules use `next/dynamic` with `ssr: false`; assert skeleton is exported.
- Render test: mount the page-level component, assert skeleton renders before the chart resolves.

## Risks & mitigations

| Risk | Mitigation |
|------|------------|
| `synchronous=NORMAL` data durability concern | Safe with WAL per SQLite docs; user data is append-mostly logs, not financial. Single-machine local-first context. |
| `mmap_size=256MB` on tiny VMs | SQLite silently falls back if not enough address space. Harmless. |
| Prepared-statement cache leak | All SQL strings are static literals or composed from a closed set; bounded by code, not user input. |
| Lazy chart causes CLS | Skeleton matches chart height; tested in render tests. |
| `Promise.all` masks errors | Use `Promise.all` not `allSettled`; first error propagates as today. |

## Decomposition into implementation slices

Eight slices, in order, all shipped together as 0.17.0:

1. **Slice 1** — DB tuning + statement cache (foundation; lowest risk, broadest impact)
2. **Slice 2** — Overview parallel + memo (builds on the faster queries from Slice 1)
3. **Slice 3** — Lazy Recharts (independent of 1 & 2; pure frontend)
4. **Slice 4** — `tokentrace doctor --timings` flag (surfaces the existing `getAnalyticsTimingReport` for before/after measurement)
5. **Slice 5** — Suspense streaming on `/` (each panel becomes its own server component wrapped in `<Suspense>` so the route streams card-by-card)
6. **Slice 6** — Scan ingestion throughput (batched transactions, prepared-statement reuse per batch, skip re-hashing unchanged files via the existing `scan_files.file_hash` table)
7. **Slice 7** — Bundle audit + targeted lazy loads (`@next/bundle-analyzer` pass, lazy-load 50KB+ deps found, drop unused imports)
8. **Slice 8** — Bug hunt on hot perf paths (multiple focused passes for N+1 queries, accidentally-quadratic loops, missing memos)

All eight land in a single 0.17.0 release per user instruction. Earlier "non-goals" section is superseded: Suspense streaming is in for 0.17.0.
